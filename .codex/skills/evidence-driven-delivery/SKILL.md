# Evidence-Driven Delivery（证据驱动交付）

本项目里 agent 干活的默认方法论。它把两套实践融成一套:
- **构建纪律**(来自 Superpowers):不急着写代码,先理清需求 → 拆成极小任务 → 红/绿 TDD → 两阶段评审 → 收尾分支。
- **验收纪律**(来自 07-quality-review 闸门):真机模拟器里实际跑、用证据说话、严守"只验不改"的边界、blocker 绝不造假。

无论是"造东西"还是"验东西",都遵守下面的核心原则,再按任务类型选择对应的轨道。

---

## 核心原则(两条轨道共享的 DNA)

1. **证据高于声明。** 不准在没有可验证证据的情况下说"做完了 / 修好了 / 通过了"。证据 = 一个真的跑过的失败→通过的测试、一张截图、一段真实 console 日志。没有证据,就当作没做完。
2. **先过闸门,再往前走。** 每个阶段都有进入/退出条件,条件没满足(设计未签字、测试不绿、validate 不是 0 error 0 warning、没有截图)就不许进入下一阶段。
3. **先想清楚,再动手。** 看到要做东西,先退一步问清楚到底要解决什么,把方案分成能读懂的小块给人确认。做验收时同理:没在真实环境复现之前,不准把静态分析结论当成缺陷。
4. **守住角色边界。** 构建角色才写产品代码;验收角色**只验不改**。两者不能混。混了就会出现"边查边改、查不清也改不对"。
5. **blocker 要诚实。** 做不到的事(比如控制不了微信开发者工具)就如实报 blocker,绝不伪造"已完成"。假的完成比真的失败破坏性大得多。
6. **小步、可回退。** 任务越小越好,改动隔离在独立分支/worktree 里,随时能丢弃重来。

---

## 选择轨道

- 任务是**造东西 / 改东西 / 加功能 / 修 bug** → 走 **Track A(构建)**。
- 任务是**验收 / 质检 / 回归 / 上线前把关** → 走 **Track B(验收闸门)**。
- 一个 workstream 可能先 A 后 B,但**同一轮里不要同时既改又验**——验收必须由一个不碰产品代码的角色独立完成。

---

## Track A — 构建(从想法到合并)

**A1. 理清需求(不要先写代码)。**
看到"我要做 X",先别建文件。用提问把模糊想法收敛成一份规格:它到底要解决什么、有哪些边界情况、不做什么(YAGNI)。把规格**分成足够短、能一口气读完的小块**拿给人确认,逐块签字。把最终规格存成设计文档。

**A2. 隔离工作区。**
设计签字后,在新分支 / git worktree 上开干。先跑一遍项目初始化,确认测试基线是干净的(已绿),再动手——否则你分不清失败是你引入的还是本来就坏的。

**A3. 写计划。**
把工作拆成**一个个 2–5 分钟能完成的小任务**。每个任务都要写明:改哪个文件(确切路径)、改成什么样、以及**怎么验证这一步成了**。计划要清楚到"一个热情但没经验、不了解项目、还讨厌写测试的初级工程师照着也能做对"。

**A4. 每个任务走红/绿 TDD。**
对**可单元测试的逻辑**(工具函数、数据转换、`request` 封装、状态计算、表单校验规则等):
1. 先写一个会**失败**的测试;
2. 跑它,**亲眼看它失败**(确认测试本身有效);
3. 写**最小**的代码让它通过;
4. 跑它,看它通过;
5. 提交。
> 注意:小程序里**纯 UI/交互**(点击 Tab、页面渲染、键盘弹出)很难用单测覆盖——这部分不在这里硬凑测试,而是留到 **Track B** 在真机里验收。逻辑层用 TDD,交互层用真机证据,二者分工。
> 在测试之前抢着写的实现代码,删掉,按 TDD 重来。

**A5. 每个任务做两阶段评审。**
- 第一阶段:**是否符合规格**?做的是不是计划里那件事?
- 第二阶段:**代码质量**?有没有更简单的写法(降复杂度)、有没有重复(DRY)、命名/边界处理是否到位?
- 标为 **critical** 的问题会**卡住进度**,必须先修。

**A6. 收尾分支。**
所有任务完成后:确认全部测试为绿 → 给出选项(合并 / 提 PR / 保留 / 丢弃)→ 清理 worktree。不绿不收尾。

---

## Track B — 质量与真机验收闸门

**B0. 先修工作流状态(如果坏了)。**
先跑状态校验(本项目用 `node .codex/skills/workstream-delegation/scripts/context.mjs validate`)。若失败:
- **只**修当前 workstream 的 `STATE.json`,用项目标准 schema 结构;
- `priority / dependencies / tags / codePaths` 从**最近一次有效状态或归档**里恢复,**不要猜**;若找不到有效来源,**报 blocker**,不要编;
- 把 `HANDOFF.md` 压缩成最新结果包,详细检查项留在运行清单(如 `RUNTIME-QA-CHECKLIST.md`);
- **不改** verifier 脚本,**不改**任何产品代码;
- 修完跑 `... refresh` 再 `... validate`,**必须达到 0 error、0 warning** 才算这步过关。
> 这步是"元数据清理",和真机验收是两件事。如果 validate 一直过不去、修不到 0/0,**立即报 blocker**,不要把整轮时间耗在这里。

**B1. 真机验收前,先"快速探测"能不能驱动开发者工具——探不通就立刻停,绝不硬刚 GUI。**

微信开发者工具是 nw.js/Chromium 桌面应用,只有打开"服务端口"后才能被 `miniprogram-automator` 连上。所以第一步永远是探测,而不是直接动手:

1. **探测(只做这一件,几秒内出结果):** 用 `miniprogram-automator` 尝试连接 IDE 的自动化端口(或先检查自动化端口文件是否存在、用 `cli auto` 带短超时探一次)。
2. **探通了 → 自动验收:** 用 `miniprogram-automator` 以代码方式启动/连接模拟器、跳页面、截图,执行 B2 清单。
3. **探不通 → 立刻报 blocker 并停下**,在 `BLOCKERS` 里给出两个选项让人来定,不要自己继续折腾:
   - **选项 A(开一次,以后自动):** 在 DevTools 里 设置 → 安全(安全设置)→ 打开"服务端口";(如未装)装好 `miniprogram-automator`;开完回复"已启用",下一轮即可全自动验收。
   - **选项 B(这轮手动):** 由人按 `RUNTIME-QA-CHECKLIST.md` 手动点一遍,agent 只负责核对并归档证据。

**明确禁止(全是过往教训):** 探测不通时,**禁止**用 SendKeys / 模拟鼠标点击 / Windows UI Automation 去戳开发者工具界面;**禁止**修改 IDE 的任何配置文件或状态文件;**禁止**靠 `--remote-debugging-port` 之类参数硬试。这些对 nw.js/Chromium 既不可靠又越界,只会白白烧掉大量时间和 token。探不通就是探不通——如实报 blocker,远比折腾二十分钟有价值。

> 不管走自动还是手动,底线不变:**静态分析永远不等于验收,必须拿到真机运行的证据。**

**B1.5 自动验收失败 3 次就封顶——别死循环,转文字带教。**
自动验收一旦连上就开始跑;但**同一步骤连续失败最多重试 3 次**。第 3 次仍失败,立刻按下面做,**不许继续无脑重跑**(同一个错误反复出现,几乎不会靠再跑一遍变好,只会白白烧掉时间和 token):
1. **停掉自动重试**,把最后一次的真实报错 / 截图记进 `BLOCKERS`;
2. **切到文字带教模式**:用清晰的分步骤教用户自己在模拟器里手动验这一片,每一步都写明——打开哪个页面、点哪里、**预期看到什么**、不符又说明什么问题(可直接引用 `RUNTIME-QA-CHECKLIST.md` 的对应项);
3. 让用户把结果(截图或一句话)回给你,你只负责**核对 + 归档证据**,不再自己去硬连。
> 计数按"同一个错误反复出现"算;真的换了不同修法才可以重置。这条等于在自动路线翻车时,自动兜底到 B1 的"选项 B(人工)",避免 agent 卡死在无限重试里。

**B2. 验收清单(按当前 slice 的范围执行,别超范围也别漏)。**
1. 清空 Console、重新编译,记录**真实**的错误和警告;
2. 点击 5 个 Tab(首页 / 交流 / 职位 / 消息 / 我的);
3. 普通路由 + 动态路由(职位、消息、候选人、岗位预览);
4. 搜索、筛选、分页、断网、超时、重试;
5. 本地登录、角色切换、隐私协议、登录失效;
6. 企业表单校验、预览、候选人管理、防重复提交;
7. 响应式:320px 窄屏、大字体、键盘弹出、模态框;
8. 截图 + 结果存到 `workstreams/<ws>/evidence/<日期>/`。
> slice 名(例如 `frontend-framework-review-v2-recheck`)决定**测多大范围**:是只复查某一片,还是整 app 全量回归。开工前先确认范围与 slice 一致,不一致就先澄清,别凭感觉扩大或缩小。

**B3. 静态结论只能算"风险",复现了才算"缺陷"。**
静态统计出来的字体、媒体查询、`adjust-position` 等问题,**没在模拟器里复现之前,不得声称是产品缺陷,也不得批量改业务页面**。复现成功 → 记为 defect;无法复现 → 记为 risk。

**B4. 证据门禁。**
**没有截图和运行证据,不得进入 REVIEW 或 SHIP。** 这是硬门槛。

**B5. 只记录,不修复。**
验收角色发现问题时,只记录:**文件 / 页面 / 复现步骤 / 截图 / 严重级别**,然后退回 owning workstream 去修。不要在验收轨道里顺手改产品代码。

---

## 通用边界(两条轨道都必须遵守)

- 验收阶段**禁止修改 miniprogram 下的产品代码**。
- **禁止**擅自启动后端 / API,或清理 mock(除非这正是任务本身)。
- **禁止**伪造完成。控制不了工具、缺少证据、依赖未就绪 → 一律如实报 blocker。
- 不准把一次授权当成长期授权;每个有副作用的动作(删除、发布、合并、提交)单独确认。

---

## 统一返回格式

每次交付都按这个结构返回(标签保持英文,方便流水线解析):

```
RESULT          —— 这轮做完/验完了什么,一句话结论
CHANGED         —— 改了哪些文件(没有就写 none)
VERIFY          —— 怎么验证的(命令、validate 结果、测试绿否)
RUNTIME EVIDENCE—— 真机证据:截图路径、console 摘要(Track B 必填)
FINDINGS        —— 发现的缺陷:文件/页面/复现步骤/截图/严重级别
RISKS           —— 仅静态发现、未复现的风险项
BLOCKERS        —— 卡住的地方(没有就写 none)
NEXT            —— 下一步建议,或退回哪个 workstream
```

---

## 严重级别定义(统一口径,避免每次标准不一)

- **blocker** —— 阻断主流程,无法继续(如核心页面白屏、Tab 点不动)。
- **critical** —— 关键功能不可用或数据错误(如登录失效后仍能操作、表单提交丢数据)。
- **major** —— 明显功能/体验缺陷,但有 workaround(如分页错乱、筛选偶发失效)。
- **minor** —— 轻微问题、文案、边角样式(如窄屏下间距不齐)。

## 一句话自检(交付前问自己)

> 我说"完成"的每一项,背后是不是都有**真的跑过的证据**?我有没有越过自己的角色边界去改了不该改的东西?有做不到的地方我是诚实报了 blocker,还是糊过去了?
